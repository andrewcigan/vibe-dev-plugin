# /upgrade-project — перевод живого проекта на актуальный движок

## Зачем

При обновлении плагина движок (`hooks/`, схема state-machine, строгость) меняется. Чтобы **живые проекты не ломались автоматически**, они остаются «legacy» (структурные проверки = warn), пока пользователь явно не выполнит апгрейд. UI-evidence gate (B2/feat-204) строг всегда — он не понижается даже для legacy.

`legacy` = в проекте нет `.harness/engine-version` или его major-версия ниже 6.

## Что делает (по шагам)

1. **Подтвердить, что это vibe-проект** — есть `.harness/` или `feature_list.json` в целевой папке.
2. **Dry-run ПЕРЕД переключением.** Прочитать текущий `feature_list.json` и проверить глазами/скриптом, что после перехода на strict он не начнёт блокироваться:
   - есть ли `passing`/`done` фичи с пустым `evidence`?
   - есть ли UI-фичи (`category: ui` или `.tsx/.vue/.svelte` в `affected_files`) в `passing` без `evidence.layer_4_user_at`/`layer_5_user_at`? — это **заблокирует** strict.
   - есть ли невалидные `state` (не из `schemas/feature-state-transitions.yaml`)?
   Если такие есть — **сначала исправить** (дописать evidence по реальным прогонам / откатить state), потом апгрейдить. Не выставлять evidence «задним числом» без реального прогона — это вернёт B2.
3. **Выполнить механику** — запустить:
   ```bash
   bash "${CLAUDE_PLUGIN_ROOT}/scripts/upgrade-project.sh" "$(pwd)"
   ```
   (ставит `.harness/engine-version=6.0`, `.harness/profile=strict`, снимает `.harness/hook-mode`).
4. **Сообщить пользователю дельту** — что теперь блокируется (UI без evidence, невалидные переходы, битый JSON, bulk без checklist) и что предупреждает (гонки записи). На бизнес-языке: «проект теперь под полной защитой — нельзя пометить экран готовым без твоего подтверждения, нельзя запустить дорогой массовый API без чек-листа».
5. **Закоммитить** изменение `.harness/` осмысленным сообщением.

## Когда НЕ применять

- Проект уже на актуальном движке (`.harness/engine-version` = 6.x) — апгрейд не нужен (скрипт идемпотентен, но смысла нет).
- Идёт активная миграция и пользователь сознательно держит learn-mode — не форсить.

## Граница

Это discipline + механика: скрипт делает переключение детерминированно, но **dry-run (шаг 2) — ответственность агента**. Нельзя переключать на strict, не проверив, что текущее состояние пройдёт — иначе следующая же запись в feature_list.json заблокируется.
