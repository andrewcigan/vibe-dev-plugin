# Финализация каждого сообщения — обязательный элемент

> Источник: реальный случай в CRM-проекте. Основатель-непрограммист: «половину вещей я не знаю, что от меня надо».

Каждое сообщение пользователю, содержащее **результат или контент**, обязано заканчиваться одним из **трёх шаблонов** ниже. Без явной финализации пользователь теряется.

## Шаблон A — у меня есть вопрос

```
**У меня есть вопрос:**
- [Конкретный вопрос 1]
- [Конкретный вопрос 2, если есть и НЕ technical]

[Если уместно: рекомендация по умолчанию — «Если ничего не скажешь — беру X».]
```

**Когда применять**: реально нужен бизнес-выбор пользователя — модель монетизации, ICP, цена, видимая функция, авторизация, данные, destructive действия.

**Не больше 1-2 вопросов**. Если ≥3 — пересобрать.

## Шаблон B — вопросов нет

```
**Вопросов от меня нет.** Следующий шаг — [конкретный шаг]. Согласен?
```

**Когда применять**: завершил работу, дальше очевидный следующий шаг pipeline.

## Шаблон C — пауза

```
**Я на паузе** — жду твоего ответа на [конкретный вопрос].
```

**Когда применять**: остановился в середине работы, нужна информация от пользователя.

---

## Anti-patterns (что НЕ делать)

- ❌ Сообщение заканчивается описанием результата без call-to-action: «Дальше — /architecture» (это анонс, не call-to-action)
- ❌ «двигаемся дальше?» без указания КУДА
- ❌ 3+ вопроса в одном сообщении (сигнал что плохо подумал — пересобрать)
- ❌ Открытый вопрос «есть ли специфика?» без рекомендации по умолчанию
- ❌ Technical A/B как вопрос (см. `rules/quality-gate.md` Check 1)

## Когда финализация НЕ требуется

- Короткие реплики в чате типа «ок, делаю» — без контента
- Прогресс-апдейты в середине длинной задачи (raw status)

## Когда финализация обязательна

- Любое сообщение с **результатом** работы (артефакт создан, проверка прошла, фича закрыта)
- Любое сообщение с **выбором** для пользователя
- Финал каждого skill (`/new-project`, `/architecture`, `/feature`, `/verify`, `/handoff`, `/ship` — все)

## Enforcement

В этом релизе — **дисциплина агента** + примеры в каждом SKILL.md.

Будущий механизм: `agents/quality-gate-validator.md` Sonnet subagent (опционально) — проверяет финальные сообщения на наличие одного из 3 шаблонов. Sources: `rules/banned-terms.yaml`.

## Глобальное правило

Это правило **дополняет** `rules/quality-gate.md` (общий фильтр коммуникации). Quality Gate описывает что **НЕ** делать. Финализация описывает что **обязательно сделать**.

## Примеры

### Хорошо (после /new-project)

```
Готово. Создал my-project, harness bootstrap (4 файла), 
режим FAST, стек — Next.js + Supabase + Claude через подписку.

**У меня есть вопрос:**
- Есть специфика ниши которую важно сразу записать в domain-rules.yaml
  (например: «работаем по СНГ», «клиенты — самозанятые», «трафик через Telegram»)?

Если ничего не скажешь — иду в /architecture. Готов?
```

### Плохо

```
✓ Проект создан в my-project
✓ Harness bootstrap: 4 файла
✓ Режим: FAST
✓ Стек: Next.js + Supabase

Дальше: /architecture — V0 архитектура + TOC bottleneck-анализ.

Прежде чем продолжить — есть ли специфика ниши?
```

Что не так:
1. «Дальше: /architecture» — анонс, не call-to-action
2. «TOC bottleneck» — жаргон без объяснения
3. Открытый вопрос без рекомендации по умолчанию
