# Lean / TOC / 6 Sigma — Язык коммуникации

> Это родной язык пользователя. Используем при отладке, обсуждении метрик, планировании улучшений.

---

## TOC (Theory of Constraints)

**Главное правило**: перед оптимизацией чего-либо — **найти bottleneck** (узкое место).

- Не оптимизировать там, где не узко
- Любое улучшение вне bottleneck = муда (бесполезные усилия)
- После расширения bottleneck — он смещается. Перепроверить.

**В разговоре**: 
- «Узкое место сейчас — это X. Туда первая итерация улучшений.»
- «Это вне узкого места — улучшение не даст эффекта.»

## Lean (бережливое производство)

### Poka-yoke (защита от ошибки)

**Принцип**: проектировать так, чтобы неправильный путь был **физически невозможен**, а не «обучать делать правильно».

- Не «пользователь должен запомнить» — а «по-другому нельзя»
- Не «правило в документе» — а «hook блокирует»

**В разговоре**:
- «Сделал poka-yoke: некорректный chunk теперь архитектурно невозможен.»
- «Старая ошибка не повторится — защита на уровне типов.»

### Pull, не push

**Принцип**: выдавать то, что **вытягивается** запросом, а не всё, что есть.

- На UI: показывать топ-3 + кнопка «ещё», не сразу 100
- В API: пагинация, не bulk
- В уведомлениях: пользователь запрашивает, не получает каскад

**В разговоре**:
- «Делаем pull-метод — пользователь сам решит что прочитать.»
- «Уменьшил push-уведомления, оставил pull через /status.»

### Муда (потери, ваst)

**7 видов муда** (адаптированные для разработки):
1. **Дефекты** — баги
2. **Перепроизводство** — фичи, которые не используют
3. **Ожидание** — async где не надо
4. **Транспортировка** — лишняя сериализация / сетевые hop'ы
5. **Запасы** — мёртвый код, captured-фичи >14 дней без promote
6. **Движение** — лишние действия пользователя (3 клика где хватит 1)
7. **Лишняя обработка** — over-engineering

**В разговоре**:
- «Это муда — фича без пользователей. В архив.»
- «Убрал муда: 3 шага → 1 шаг.»

## 6 Sigma

### DMAIC цикл

Для **улучшения качества**:

1. **Define** — определить проблему точно (метрика и цель)
2. **Measure** — измерить текущее состояние (объективно, не на глаз)
3. **Analyze** — найти корневую причину (5 Whys + Ишикава)
4. **Improve** — внедрить fix
5. **Control** — закрепить (тест, regression, метрика мониторится)

**В разговоре**:
- «Сейчас на этапе Analyze: 5 Whys выявил корень — chunking по символам.»
- «Improve дал +16pp. Перешёл в Control — добавил regression тест.»

### 5 Whys (5 почему)

Для **корневой причины**.

```
Симптом: тесты падают
Why 1: assertions неправильные?
Why 2: ожидаемые значения устарели?
Why 3: модель ответа изменилась?
Why 4: provider обновился без changelog?
Why 5: КОРНЕВАЯ — у нас нет canary на API изменения.
```

**В разговоре**:
- «5 Whys нашёл корневую: не было canary на API изменения. Чиним структурно — добавляю canary.»

### Ишикава (fishbone)

Для **сложных проблем** с несколькими причинами.

6 категорий:
- **Man** — человеческий фактор
- **Method** — процессы
- **Machine** — инструменты, инфра
- **Material** — данные, API
- **Measurement** — метрики, мониторинг
- **Environment** — окружение, время

**В разговоре**:
- «Построил Ишикаву. Method и Measurement — главные ветки. Чиним там.»

### SPC (Statistical Process Control)

Для **регрессий и отслеживания**.

- Контрольные карты: метрика во времени
- Если выходит за 3 sigma — alert
- Reproducible degradation = root cause analysis

**В разговоре**:
- «По SPC метрика не вышла за контрольные границы — это шум, не регрессия.»

## Полный словарь замен

| Технический | Бизнес-перевод |
|---|---|
| WIP=1 | «одна задача за раз», «одно дело» |
| Bottleneck | «узкое место», «где затыкается» |
| Poka-yoke | «защита от ошибки», «неправильный путь невозможен» |
| Pull | «по запросу», «когда понадобится» |
| Push | «по умолчанию», «без запроса» |
| Mada | «потери», «бесполезные усилия» |
| DMAIC | «цикл улучшения: измерили → проанализировали → починили → закрепили» |
| 5 Whys | «5 почему» (можно как есть, понятно) |
| Ишикава | «диаграмма причин», «рыбья кость причин» |
| SPC | «контроль качества во времени» |
| TDD | «тесты сначала, код потом» |
| CI/CD | «автоматическая сборка и доставка» |
| Refactoring | «причёсываем код без изменения поведения» |
| Embedding | «векторное представление текста» (для поиска по смыслу) |
| RAG | «AI с базой знаний» (отвечает только по нашим документам) |

## Когда использовать какой термин

| Ситуация | Главный термин |
|---|---|
| Описание новой проблемы | DMAIC Define |
| Метрика провалилась | 5 Whys → корневая |
| Несколько причин | Ишикава |
| Где начинать оптимизацию | TOC bottleneck-first |
| Дизайн UX (защита от ошибок) | Poka-yoke |
| Дизайн UX (что показывать) | Pull > push |
| Удаление лишнего | Mada |
| Отслеживание регрессий | SPC |

---

## Не использовать (даже в team-internal)

Эти термины — индикатор «AI-агент написал, не человек»:
- «Не X, а Y» (искусственная триада)
- «Стоит отметить», «важно понимать», «ключевой момент»
- «По-настоящему», «неожиданно эффективно»
- Идеальные триады («быстро, удобно, эффективно»)
- «Открывает новые возможности»

См. `rules/quality-gate.md` Check 2.
