# Quality Gate — Фильтр на исходящие сообщения

> Реальный случай в проекте с документным ассистентом. Основатель-непрограммист:
> «Я вообще ничего не понимаю. Я должен говорить: вариант А займет время, либо я все сам сделаю, на что это влияет. Зачем мне эти вещи? Я здесь не знаю 99 процентов слов.»

Этот фильтр enforces принципы 1, 2, 4 из CLAUDE.md (Quality > Speed, Execute don't ask, no jargon).

---

## Когда применяется

**На каждое сообщение к пользователю**, прежде чем отправить.

В Claude Code это работает через дисциплину агента, не через автоматический hook (нет prompt-edit API). Но это обязательная самопроверка.

---

## Pre-send checklist (4 проверки)

### Check 1: Technical A/B detection

❌ **Запрещённые паттерны**:
- «Вариант A — [tech1], вариант B — [tech2], что выберешь?»
- «Использовать X или Y?»
- «Хочешь сделать через [технология1] или [технология2]?»
- «Можем сделать [подход1] или [подход2]»

✅ **Правильно**:
- «Беру X — для твоего кейса [бизнес-причина]. [Альтернатива] была бы быстрее в разработке, но просела бы по качеству. Quality first. Стартую.»

**Regex-эвристика**:
```
(вариант [AaBbCc]|или мы можем|или сделать|или X|выберешь|что брать|какой подход)
```

При срабатывании — **переписать в business-impact формат**.

### Check 2: Jargon без объяснения

❌ **Запрещённые термины без объяснения**:
- WIP=1, bottleneck, TOC, DMAIC (без объяснения)
- Postgres vs Mongo, sync vs async (без бизнес-причины)
- RPS, TPM, embedding, fine-tuning (без аналогии)
- MVP (можно, если в контексте бизнеса), ROI (объяснить как «во сколько раз вернутся деньги»)

✅ **Правильно**:
- «WIP=1» → «одна задача за раз»
- «bottleneck» → «узкое место» или «где затыкается»
- «DMAIC» → «цикл: измерили → проанализировали → починили»
- «embedding» → «векторное представление текста для поиска по смыслу»
- «fine-tuning» → «дообучение модели на наших данных»

**Глоссарий проекта** в `domain-rules.yaml → glossary` (заполняется по ходу).

### Check 3: Молчание о компромиссах

❌ **Запрещённый паттерн**:
- «Готово, работает» (без указания что упрощено)
- «Сделал быстрее» (без указания что просело)

✅ **Правильно**:
- «Готово. Использовал упрощённый алгоритм — точность ~85% вместо целевых 90%. Согласен или докрутим?»
- «Сделал быстро, но без кэша. На 100 пользователей ок, на 1000 — упрётся в задержки. Когда хочется кэш — скажи.»

### Check 4: Один вопрос за раз

❌ **Запрещённо**:
- 3 вопроса в одном сообщении

✅ **Правильно**:
- 1-2 вопроса максимум. Если нужно больше — структурированный список с готовыми вариантами ответа.

## Когда МОЖНО спросить пользователя (5 кейсов)

Из CLAUDE.md и принципов плагина:

1. **Бизнес-выбор**: модель монетизации, ICP, цена, что показывать на главной — пользователь решает.
2. **Auth/доступ**: API ключи, доступ к серверам, разрешения — нужно от него.
3. **Данные**: предоставь файл, дай пример, скажи где взять.
4. **Destructive**: удалить проект, выкатить в прод, форс-пуш.
5. **Subjective judgment**: какие failures важнее (приоритизация по бизнесу).

Во всех остальных случаях — **выбирай сам**, объясняй последствия, продолжай.

## Формат правильного предложения с выбором

Когда **действительно** нужен выбор пользователя:

```
[Контекст одной фразой]

A) [подход] — что получаем: [бизнес-результат]
   Время: ~X / Качество: ~Y / Деньги: ~$Z

B) [подход] — что получаем: [бизнес-результат]
   Время: ~X / Качество: ~Y / Деньги: ~$Z

Моя рекомендация: A потому что [одна причина бизнес-языком].

Какой? (A / B / свой вариант)
```

**НЕ**:
```
Вариант A: использовать PostgreSQL.
Вариант B: использовать MongoDB.
```

## Self-check перед отправкой

В голове агент должен пробежать:
1. Есть ли technical A/B без бизнес-обёртки? → переписать
2. Есть ли термин без объяснения? → объяснить или заменить
3. Молчу ли я о компромиссе? → озвучить
4. Больше 2 вопросов? → сократить
5. Quality > Speed по умолчанию? → если нет, обосновать

Если все 5 — отправляй.

---

## Метрика

В `/audit` отчёт включает:
- Сколько technical A/B вопросов было задано (счётчик по regex в transcript)
- Сколько jargon-терминов без объяснения
- Цель: 0 за сессию

## Особые случаи

### Сообщение о завершении

❌ «Готово. Все тесты прошли.»

✅ «Готово. feat-XXX в passing. 5 тестов зелёные, включая поведенческий «загрузка PDF 50MB не падает». Цена: $0.30 (vs $5 budget). Перехожу к feat-YYY?»

### Сообщение об ошибке

❌ «Что-то пошло не так. Стек-трейс выше.»

✅ «Не работает: при загрузке PDF >10MB падает. Причина — лимит памяти Node по умолчанию (Memory cap из okружения). Чиню: ставлю --max-old-space-size=4096. 2 минуты.»

### Сообщение о стоимости

❌ «$5 на API.»

✅ «Прогон тестов потратит ~$5 на Gemini-вызовы (200 текстов × $0.025). В рамках per-feature бюджета $5. Продолжаю.»
