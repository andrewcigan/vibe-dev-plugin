# Карпати-принципы (Микро-слой методологии)

Применяются агентами **на каждое действие** (Edit / Write / Bash / Read).

---

## 1. Think Before Coding

Перед написанием кода — **остановиться** и ответить на 3 вопроса:

1. **Что решаем?** (точная формулировка цели)
2. **Какие интерпретации?** (2-3 варианта что это может значить)
3. **Какие альтернативы?** (2-3 разных подхода)

Записать в SESSION.md секцию `Decisions Made` или в `implementation-notes`.

**Анти-паттерн**: «надо сделать X» → сразу пишу код.

---

## 2. Simplicity First

Начинаем с **простейшего работающего решения**. Сложность обосновывается данными (метрики, требования), не воображением.

- Если 3 if'а решают задачу — не строить Strategy pattern
- Если RecursiveCharacterTextSplitter даёт 80% — не лезть в semantic chunking
- Если SQL даёт нужный ответ — не делать ORM

**Запись в decisions**: почему выбрали простое (или почему пришлось сложное).

**Анти-паттерн**: over-abstraction, premature optimization.

---

## 3. Surgical Changes

Трогаем **только то, что нужно для задачи**. Никакого drive-by refactoring «заодно».

- Diff должен касаться только `feature.affected_files`
- Заметил что-то другое плохое → запиши в backlog как новую фичу, не чини мимоходом
- Pre-commit hook блокирует scope-leak

**Запись**: если что-то плохое заметил — `backlog/captured` в feature_list.json.

**Анти-паттерн**: «заодно почистил соседний файл».

---

## 4. Goal-Driven Execution

Цель = **критерии успеха**, не список шагов.

- НЕ: «выполнил 5 шагов инструкции — готово»
- ДА: «verification_command зелёная — готово»

Цикл, пока цель не достигнута. Не «сделал 3 изменения и хватит».

**Анти-паттерн**: чек-лист исполнения вместо итерация до цели.

---

## 5. Test-First Reproduction (для багов)

Прежде чем фиксить баг — **воспроизвести его тестом**.

1. Описание бага → red test
2. Запустить — должен упасть с правильной симптоматикой
3. Только потом — fix
4. Тест должен стать зелёным

Если не получилось воспроизвести в тесте — **диагноз не подтверждён**. Не чинить вслепую.

**Запись**: red test первым коммитом, fix вторым.

**Анти-паттерн**: «попробую так — может починилось».

---

## 6. Hidden Assumptions Elicitation

Перед стартом сложной задачи — **проговорить допущения**.

- «Я предполагаю что пользователь знает свой номер телефона»
- «Я предполагаю что Telegram webhook доставляется в течение 1 секунды»
- «Я предполагаю что данные не повторяются в одной сессии»

Записать в `domain-rules.yaml` (если про продукт) или `docs/decisions/` (если про техническое).

Дать пользователю возможность изменить (один вопрос максимум).

**Анти-паттерн**: молча додумать, потом «ой, не работает потому что Y».

---

## Применение по типу действия

| Действие | Что обязательно из 1-6 |
|---|---|
| Bash (read-only): ls, cat, grep | — |
| Bash (writing): mkdir, rm, mv | 3 (surgical) |
| Bash (bulk API): curl × N | 6 (hidden assumptions: квоты, batch, task_type) + pre-launch-checklist |
| Read | — |
| Edit (existing) | 1, 3 (think, surgical) |
| Write (new file) | 1, 2 (think, simplicity) |
| Fix bug | 5 (test-first) |
| New feature | 1, 2, 3, 6 (think, simplicity, surgical, assumptions) |
| Refactor | only if в активной фиче AND обоснован метрикой (не drive-by) |

---

## Самопроверка перед каждым tool call

Агент должен пробежать:
1. Я подумал перед действием? (1)
2. Это самое простое решение? (2)
3. Diff минимальный? (3)
4. Я двигаюсь к цели или к чек-листу? (4)
5. Если баг — есть red test? (5)
6. Я записал assumptions? (6)

Если хотя бы один «нет» без обоснования — STOP, добавь обоснование или переделай.

## Связь с другими слоями

- **Слой 0 (Enforcement)**: pre-commit hook принуждает к surgical (#3); init.sh принуждает к Test-First через verification structure (#5)
- **Слой 2 (Lean/TOC)**: Simplicity First (#2) согласуется с муда-elimination
- **Слой 3 (владелец продукта)**: Goal-Driven (#4) — это валидационная выборка ≥90% как gate
- **Слой 4 (Communication)**: Think Before (#1) — это варианты A/B/C в Quality Gate format
