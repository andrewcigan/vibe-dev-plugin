# Vibe Dev v5.1 — Методология

4 слоя дисциплины, наложенные друг на друга. Каждое действие агента проходит через все 4.

---

## Слой 0: Harness as enforcement (НОВОЕ в v5.1)

См. `workflow/enforcement-philosophy.md`.

**Каждый принцип имеет**:
1. Файл где зафиксирован
2. Механизм enforcement (hook / agent / schema)
3. Что произойдёт при попытке обхода

Без всех 3 — это не инвариант, а пожелание.

---

## Слой 1: Микро (на каждый Edit/Write/Bash) — Карпати

Применяется агентами **на каждое действие**.

1. **Think Before Coding** — «что решаем? 2-3 интерпретации? какие альтернативы?»
2. **Simplicity First** — простейшее работающее решение; сложность обосновывается данными
3. **Surgical Changes** — трогаем только нужное; no drive-by refactoring
4. **Goal-Driven Execution** — цель = критерии успеха, не список шагов
5. **Test-First Reproduction** — баг → тест → fix (не наоборот)
6. **Hidden Assumptions elicitation** — записать в `domain-rules.yaml` или `docs/decisions/`

**Эффект**: отсутствие over-engineering, drive-by refactoring, додумывания, features-beyond-the-ask.

См. `rules/karpathy-principles.md` + `rules/anti-patterns.md`.

---

## Слой 2: Мезо (на каждую фичу/этап) — Lean / TOC / 6 Sigma

Применяется агентами **при планировании и проектировании**.

- **TOC**: перед оптимизацией — найти bottleneck
- **Lean**: poka-yoke в UX, pull-методы, муда-elimination
- **6 Sigma**: DMAIC для улучшения, 5 Whys для root causes, Ишикава для сложных проблем, SPC для регрессий

**Эффект**: фокус усилий на правильных местах; системный подход; родной язык пользователя.

---

## Слой 3: Макро (на весь проект) — Авторская методология

Канонический цикл:
1. Long-list гипотез (10+, аргументированных, со ссылками)
2. Критический subagent → отсев / доработка
3. Параллельные ресёрчи (GitHub / форумы / arxiv / reddit)
4. Объединение результатов (synthesizer)
5. Реализация с тестами (Test-First)
6. Экспертная оценка (другой LLM / другая модель) → возврат на доработку
7. Валидационная выборка ≥90% → передача пользователю

**Эффект**: проекты доходят до реального качества, не «выглядит работающим».

Этот цикл встроен в FAST/FULL pipeline через этапы.

---

## Слой 4: Communication layer — Портрет пользователя

Применяется системой **на каждое исходящее сообщение пользователю**.

- **Quality Gate** (см. `rules/quality-gate.md`) — фильтр на технические A/B, термины без объяснения, молчание о компромиссах
- **Бизнес-язык**: «один заказ за повара» вместо «WIP=1», «узкое место» вместо «bottleneck»
- **Формат предложений**: «беру X, потому что Y, остановить?» вместо «вариант A или B?»
- **Показывать**: концепции, схемы, feature list, trade-offs, метрики, команды
- **Скрывать**: реализационный код, имена/форматирование, выбор эквивалентных библиотек, стек-трейсы

## Как слои взаимодействуют (пример)

**Задача**: «нужен Telegram-бот для бронирования столов».

**Слой 4 (Communication)**:
- Спрашиваю ОДНО: «Telegram + что в ресторане? (имя, география, тип кухни — три вопроса в одном)»
- Не спрашиваю technical (Postgres vs Mongo)

**Слой 3 (Авторская методология)**:
- /research если FULL (как тестируют похожие боты, что есть на рынке)
- Long-list 10 фичей → critique → 5 для MVP

**Слой 2 (Lean/TOC)**:
- Bottleneck: распознавание дат на естественном языке («на сегодня вечером», «не на этой неделе»)
- Poka-yoke: невозможно выбрать прошедшую дату
- Mada: убрать sms-уведомления (избыточны при Telegram)

**Слой 1 (Карпати)**:
- Каждая фича: Test-First, simplest working, surgical
- Hidden assumption: «пользователь знает свой номер» → записать в assumptions, спросить

**Слой 0 (Enforcement)**:
- WIP=1 enforced через pre-commit hook
- AGENTS.md ≤200 строк через init.sh check
- Cost preview перед bulk API
- 4-layer verification обязательна

## Когда применять что

| Задача | Главный слой | Дополнительные |
|---|---|---|
| Маленький фикс | Слой 1 (Карпати) | Слой 0 (WIP=1, scope-check) |
| Баг | Слой 1 (Test-First) | Слой 2 (5 Whys) |
| Улучшить качество | Слой 3 (Авторская методология) | Слой 2 (DMAIC) |
| Оптимизация | Слой 2 (TOC bottleneck-first) | Слой 1 |
| Новый UX | Слой 2 (Lean) | Слой 4 (показать концепт, скрыть код) |
| Сложный сбой | Слой 2 (5 Whys + Ишикава) | Слой 3 (LLM-кворум при stuck) |
| Новый проект | Слой 3 (полный цикл) | Все остальные внутри |
| Регрессии | Слой 1 (Test-First) | Слой 2 (SPC) |

## Пример сообщения с правильным сочетанием слоёв

**Неправильно** (только Слой 1):
> «Поменял chunking, теперь работает лучше.»

**Правильно** (все 4 слоя):
> «По DMAIC: измерили качество 72%. Через 5 почему нашли корневую причину — chunking по символам вместо абзацев. Исправили (poka-yoke — теперь некорректный chunk невозможен по архитектуре). Новый score 88%. Узкое место сместилось на embedding latency — следующая волна по нему. Стоимость: $3/индекс. Согласен?»

Это:
- Слой 2: DMAIC, 5 почему, poka-yoke, bottleneck
- Слой 1: chunking исправлен surgical
- Слой 4: бизнес-язык, концепция выше деталей, бюджет, явный вопрос «согласен?»
- Слой 0: enforcement через архитектуру (некорректный chunk невозможен), не через документацию
