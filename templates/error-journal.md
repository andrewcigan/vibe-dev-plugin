# Error Journal — Live Log of "Не работает" + Fixes

> Этот файл — подсистема **Learning** (одна из 7). Создаётся при первом «не работает» от пользователя или системы.
> Цель: каждая ошибка платит за себя один раз — больше никогда не повторяется.

## Когда сюда писать (маршрутизация — C3)

- **Сюда:** любое «не работает» (от пользователя или системы), тупик, ОТВЕРГНУТАЯ попытка (что не сработало и почему), найденная корневая причина, рекуррент.
- **НЕ сюда:** архитектурные решения → `docs/decisions/`; прогресс/состояние → `SESSION.md`; готовые уроки для памяти → `feedback_*.md` (промоушен при `/handoff`).
- **Чтение по умолчанию — только записи со `status: active`.** Устаревшее НЕ стираем — помечаем `status: superseded/expired` (см. штамп ниже). Так журнал не врёт и не теряет историю.

## Триггеры автоматической записи

**От пользователя (агент слушает ключевые фразы):**
- «не работает», «не пашет», «не так», «не то»
- «опять», «снова», «уже было»
- «ошибка», «проблема», «ломается»
- «переделай», «перепиши», «удали»
- «криво», «халтура»
- «не понимаю», «не вижу», «не нахожу»

**От системы (автоматические):**
- Verification fail
- Test fail (3+ раза)
- Stuck-protocol fires
- Auto-rollback
- Cost overrun
- Concurrent-write block

Каждая ошибка пишется в формате ниже. **ID** автоинкрементный.

```
## err-NNN | YYYY-MM-DD HH:MM | feat-XXX

**status**: active            # active | superseded | expired — по умолчанию читаем только active (G1)
**recorded_at**: YYYY-MM-DD HH:MM
**superseded_by**: —          # id записи, которая это заменила (заполнить при status: superseded)

**Курсор (форма против повторов — C2):**
- problem: [в чём проблема одной фразой]
- attempted: [что уже пробовали]
- rejected: [что НЕ сработало и почему — это тупики, не повторять]
- next_step: [следующий шаг / где остановились]

**Триггер**: [что сказал пользователь / что произошло автоматически]
**User reported**: yes / no
**Verification status at time**: passing / failing / not_run
  ↑ КРИТИЧНО: если passing — у нас плохие verification commands, надо улучшать

**Контекст**: [одна фраза о ситуации]

**5 Why анализ**:
1. Почему это случилось? → ...
2. Почему? → ...
3. Почему? → ...
4. Почему? → ...
5. Корневая причина: ...

**Класс ошибки**: [api_research / concurrent_write / domain_knowledge / tool_use / cost_estimation / scope_leak / state_drift / ...]

**Fix**: [конкретно что изменили]
**Verification after**: [как убедились что починили]

**Урок**: [одно предложение — что больше не повторять]
**Lesson scope**: project / user / system
**Promoted to memory**: yes/no → [path/to/feedback_*.md]

**Recurrence check**: [совпадает ли с err-X из прошлой истории? Если да — это РЕКУРРЕНТ, критическая дыра]
```

## Recurrence Tracking

При записи новой ошибки система проверяет: совпадает ли её **класс** с уже зафиксированными?

- Если **0 совпадений** — норма
- Если **1+ совпадение** — Recurrence Alert: «ОПЯТЬ ЭТО — было err-X N сессий назад. Прочитай прежде чем продолжать»
- **Recurrence rate** = % ошибок, повторившихся хотя бы раз. **Цель: 0%.**

Это метрика подсистемы Learning. Показывается в `/audit`.

## Pipeline по фазам сессии

### Live (во время сессии)
- Триггер → запись за 5 секунд
- Recurrence check → блок «было раньше»

### При `/handoff`
- Группировка похожих ошибок этой сессии
- Каждой назначается lesson_scope
- Предложение пользователю одной строкой: «3 ошибки → 2 урока в память. Confirm? (д/н/edit)»
- При confirm → promotion:
  - `project` → `.claude/projects/<this>/memory/feedback_*.md`
  - `user` → `.claude/memory/MEMORY.md` (или MEMORY.md проекта если связано с подходом к работе)
  - `system` → предложение добавить в глобальный CLAUDE.md

### При `/ship` или milestone
- Полная ретроспектива в `./retrospectives/YYYY-MM-DD-<project>/`
- Топ-3 повторяющиеся ошибки
- Что зафиксировано в память, что не зафиксировано

### При `/audit`
- Метрики: error velocity, recurrence rate, top 3 root-cause classes
- Если recurrence_rate > 0% — это сигнал что Verification или Learning подсистемы недостаточны
